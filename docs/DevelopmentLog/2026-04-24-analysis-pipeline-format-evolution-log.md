# 2026-04-24 分析链路输入格式演进开发日志

## 1. 背景

这一轮优化的核心目标，不是单纯“修改输入格式”，而是降低 RK3588 板端从视频输入到首次有效跌倒识别之间的总体时延。

问题最初暴露在分析链路上：虽然系统能够正常完成预览、分析和分类，但端到端性能不稳定，而且不同输入格式路径对 `health-falld` 的 CPU 压力差异很大。随着后续几轮实验推进，这条链路先后经历了：

1. 原始 `JPEG` 分析输入方案
2. 中途尝试的 `NV12` 直送方案
3. 最终收敛的 `GStreamer` 侧预处理 + `RGB` 输入方案

从结果来看，这并不是一次线性成功的优化过程，中间确实出现过明显的“反向优化”。

## 2. 第一阶段：原始 JPEG 方案

最初的分析链路采用 `JPEG` 路径：

- `health-videod` 负责产出用于分析的 `JPEG` 帧
- `health-falld` 接收这些 `JPEG` 帧
- 在 `health-falld` 内部进行解码，并转换到姿态模型需要的输入格式

这一版的优点是：

- 路径已经跑通，工程稳定性较好
- 首次分类时延虽然不算特别理想，但整体还处于可接受范围
- 预览链路与分析链路都围绕既有 `MJPEG/JPEG` 机制构建，工程复杂度相对低

这一版的缺点是：

- `JPEG` 编解码本身会引入额外成本
- 在分析端仍然存在图像解码和格式转换工作
- 后续如果继续压缩端到端时延，这条路径的上限比较有限

从后续 A/B 测试文档回看，当时部署在板子上的 `main` 基线数据为：

- `startup_classification_latency_ms = 2160`
- `classification_stage_latency_ms = 2153`

这组数字后来成为后续优化的参考基线。

## 3. 第二阶段：NV12 直送方案，以及为什么会出现反向优化

在后续优化中，曾经尝试过一条看起来更“原始”、理论上也更高效的路径：尽量绕开 `JPEG`，直接把 `NV12` 输入分析模块。

当时这条思路的出发点很自然：

- 既然 `JPEG` 有编解码开销，那就直接传更原始的视频帧
- 如果减少压缩和解码步骤，理论上应该更快

但板端实测表明，这一版不是优化，而是退化。

### 3.1 退化的根因

问题并不在于 `NV12` 这个格式本身，而在于谁来承担后续预处理成本。

在这条方案里，虽然前端链路更“原始”了，但 `health-falld` 并不能直接高效消费这份输入，于是后续大量图像预处理工作被挪到了 CPU 侧，包括：

- `NV12 -> RGB`
- resize
- letterbox

这些操作逐像素、逐帧执行，而且发生在 `health-falld` 的热点路径中，直接带来了明显的 CPU 压力和时延回退。

从当时的设计记录看，这一版候选方案在板端观测到的典型问题包括：

- 首次有效分类大约退化到 `4.2s`
- `health-falld` 内部 pose preprocess 大约达到 `60 ms/frame`

这意味着虽然绕开了 `JPEG`，但把更重的图像处理负担搬进了分析进程，最终总账反而更差。这就是那次“反向优化”的本质原因。

### 3.2 这次失败带来的关键认识

这次回退让方向变得更清楚了：

- 问题不只是“不要用 JPEG”
- 真正重要的是“不要让 health-falld 在 CPU 上做重图像预处理”

也就是说，优化重点应该是把模型输入前的颜色空间转换、缩放、整形等工作，尽量从 `health-falld` 热路径中剥离出去。

## 4. 第三阶段：收敛到 GStreamer 侧预处理 + RGB 输入

在经历 `NV12` 路径回退之后，方案才逐步收敛到更合理的方向：

- UI 预览链路继续保持原有 `MJPEG` 分支，不动现有预览能力
- 在 `health-videod` / `GStreamer` 的分析分支中，提前完成分析所需的图像预处理
- 直接输出尺寸固定、适合模型消费的 `RGB` 原始帧
- `health-falld` 只保留一条更薄的 `RGB` fast path，避免重复做 CPU 侧图像转换

这一版的本质变化不是“把格式从 JPEG 改成 RGB”这么简单，而是：

1. 把预处理工作尽量前移到多媒体 pipeline
2. 让 `health-falld` 收到更接近模型输入形态的数据
3. 从架构上移除 `health-falld` 中那段昂贵的 `NV12 -> RGB + resize + letterbox` 热点路径

从结果来看，这才是这轮优化真正成功的关键。

## 5. 最终相对 JPEG 原方案的优化幅度

根据当时的板端验证结果文档：

- `main` / 原始 `JPEG` 方案：
  - `startup_classification_latency_ms = 2160`
  - `classification_stage_latency_ms = 2153`
- 收敛后的 `RGB` 方案：
  - 一次主对比结果为 `startup_classification_latency_ms = 1315`
  - 在清理磁盘、恢复正常 `start.sh` 流程后，进一步测到 `startup_classification_latency_ms = 1259`

因此，相对 `JPEG` 原方案，最终 `RGB` 方案的改善幅度大致是：

### 5.1 首次分类启动时延

- `2160 -> 1315 ms`
  - 优化 `845 ms`
  - 幅度约 `39.1%`

如果按后续更好的那次正常启动流程结果计算：

- `2160 -> 1259 ms`
  - 优化 `901 ms`
  - 幅度约 `41.7%`

### 5.2 分类阶段时延

- `2153 -> 1399 ms`
  - 优化 `754 ms`

或者按后续正常启动流程结果：

- `2153 -> 1362 ms`
  - 优化 `791 ms`

可以认为，这次最终成功收敛后的 `RGB` 方案，相对最初 `JPEG` 方案，在首次有效分类时延上大约优化了 `0.85s ~ 0.90s`，提升幅度约 `39% ~ 42%`。

## 6. 为什么 RGB 方案最终成功

从验证文档中的阶段性结果看，最终成功并不是因为 `RGB` 天生比 `JPEG` 或 `NV12` 更“高级”，而是因为它更适合当前这条工程链路的职责划分。

它成功的关键在于：

- `health-videod` 更擅长借助 `GStreamer` 做图像格式转换和缩放
- `health-falld` 更应该聚焦在推理和分类，而不是承担重图像预处理
- `RGB` 方案把模型输入整形成一条更薄、更直接的路径

从当时的姿态阶段统计结果看：

- `avg_preprocess_ms` 已经接近 `0.0`

这说明先前那段导致退化的 CPU 预处理热点，确实已经从生产路径中基本移除。

## 7. 本轮演进的经验总结

这次开发过程里最值得记录的，不只是“最后变快了”，而是中间有一次很典型的架构误判：

- 直觉上更原始的数据格式，不一定意味着系统整体更快
- 如果新路径把更多逐像素处理压进 CPU 热路径，就可能出现明显反向优化
- 真正有效的优化，必须看整条链路的职责划分，而不是只盯着某一个局部格式选择

这次演进最终说明：

1. `JPEG -> NV12` 并不自动等于优化
2. 只有当图像预处理责任被合理前移，`health-falld` 热路径被瘦身后，新的输入格式才真正发挥价值
3. 最终的 `GStreamer` 侧预处理 + `RGB` fast path，才是这轮架构优化真正稳定有效的收敛点

## 8. 后续判断

到 `RGB` 方案收敛完成后，这条链路上的主要瓶颈已经不再是输入格式本身，而更偏向：

- 下游推理执行
- 分类阶段预算
- 更细粒度的分析链路调度与传输成本

也正因为如此，后面才会继续推进共享内存传输等进一步优化工作。

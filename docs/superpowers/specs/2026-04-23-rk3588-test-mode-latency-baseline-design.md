## RK3588 Test-Mode Latency Baseline Design

Date: 2026-04-23
Status: Approved for planning

## 1. Goal

Establish a reliable board-level latency baseline for fall-analysis startup in test mode, using `main` as the reference implementation and the current worktree as the candidate implementation.

This design does not attempt to solve the latency problem directly. Its immediate goal is to measure the gap precisely enough that the next implementation phase can target the right bottleneck.

## 2. User-Approved Scope

The user explicitly approved the following scope:

- use current `main` as the old-engineering baseline
- use the current worktree as the new implementation under test
- run both on the same RK3588 board
- use the same test input file: `~/Videos/video.mp4`
- use the existing test-mode path, not a separate ad hoc harness
- define the timing start as the moment `health-videod` successfully enters `test_file` playback
- define the timing end as the first emitted classification result such as `monitoring` / `stand` / similar first-state output after a person is present
- first-stage optimization goal: make the current worktree "not worse than main"

The user also clarified an important business constraint:

- the video begins with a person already visible, so startup-classification latency is the primary metric

## 3. Problem Statement

The current worktree is subjectively much slower than `main` during test-mode playback, even after the recent NV12 analysis-path optimization and lag/backlog fixes.

At this point, directly proposing more code changes would be premature because the observed regression may come from multiple different layers:

- delayed start of the test-file playback path in `health-videod`
- slower first analysis-frame delivery from `health-videod` to `health-falld`
- slower startup classification behavior inside `health-falld`
- different frame dropping or startup buffering behavior between `main` and the current worktree
- sequence-window or first-result policy differences that were previously masked by the older JPEG-derived path

Because the product complaint is "new engineering feels much slower than old engineering," the correct next step is a strict A/B comparison under the same board conditions.

## 4. Success Metric

### 4.1 Primary metric

For each run, measure:

```text
startup classification latency = first classification timestamp - test playback start timestamp
```

Where:

- `test playback start timestamp` = the moment the board-side video service enters `test_file` playback for `~/Videos/video.mp4`
- `first classification timestamp` = the first classification output generated by `health-falld` for that run, including bootstrap states such as `monitoring`, `stand`, or equivalent first visible-person state

### 4.2 First-stage acceptance rule

The current worktree passes the first-stage goal only if:

```text
startup classification latency(worktree) <= startup classification latency(main)
```

The comparison must be made on the same board and with the same test file and start procedure.

### 4.3 Supporting breakdown metrics

To make root-cause analysis possible, also collect:

```text
analysis ingress latency = first analysis-frame ingress timestamp - test playback start timestamp
classification stage latency = first classification timestamp - first analysis-frame ingress timestamp
```

These supporting metrics make it possible to decide whether the regression is dominated by:

- the video/analysis export side
- the fall-analysis consumer side
- or both

## 5. Test Design

### 5.1 Compared builds

Two builds will be tested:

1. `main`
   - role: baseline
2. current worktree
   - role: candidate

Both must be deployed to the same RK3588 runtime environment one at a time.

### 5.2 Shared runtime conditions

Each run must use the same conditions:

- same RK3588 board
- same camera-independent test mode
- same file: `~/Videos/video.mp4`
- same service startup flow
- same runtime mode (`system`)
- same output sockets and bundle script flow

The board must be cleanly stopped between runs so stale processes or socket state do not contaminate the comparison.

### 5.3 Start and end event definitions

#### Start event

The official start event is the board-observed moment when test playback begins for `~/Videos/video.mp4`.

The implementation phase may realize this using one of the following equivalent signals:

- a `health-videod` log line emitted exactly when `start_test_input` becomes active
- an explicit timestamp recorded by `VideoService` when the test-input state transitions to `test_file`
- another board-local signal that is equally precise and unambiguous

The key requirement is consistency between `main` and the worktree.

#### End event

The official end event is the first classification message emitted by `health-falld` after the run starts.

This includes first-available visible-person states such as:

- `monitoring`
- `stand`
- other equivalent non-empty first classification states

The end event is intentionally not limited to fall-only states, because the user wants the first useful model output after a person appears.

## 6. Recommended Measurement Approach

### Recommended approach: automated board-side A/B latency harness

Use a small, repeatable board-side measurement flow that starts the backend stack, enables test mode with `~/Videos/video.mp4`, captures timestamps for the start event and first classification event, and prints a structured result for each build.

Why this is the recommended approach:

- it compares `main` and the worktree under identical conditions
- it removes subjective timing judgment
- it produces data that can be re-run after each optimization attempt
- it gives enough breakdown detail to locate the regression boundary

### Alternatives considered

#### Manual UI / log observation only

Pros:

- fastest to begin

Cons:

- not precise enough for engineering decisions
- hard to separate playback-start delay from classification-start delay
- not repeatable enough for regression tracking

#### End-to-end total latency only, no stage breakdown

Pros:

- simple metric
- directly matches the first-stage user complaint

Cons:

- if the worktree is slower, it will not tell us where the extra delay is introduced

The stage-breakdown design is therefore preferred.

## 7. Required Instrumentation Principles

Instrumentation added in the next phase must obey these rules:

1. Use the same measurement semantics for `main` and the worktree
2. Avoid changing business behavior just to measure latency
3. Prefer explicit timestamped log or IPC events over indirect inference
4. Keep the measurement path removable or low-risk after debugging
5. Ensure one run cannot reuse old sockets, stale subscribers, or buffered classification output from a previous run

## 8. Likely Outcomes and Interpretation

### Outcome A: worktree is slower before first analysis ingress

Interpretation:

- regression is likely on the `health-videod` side
- examples: test-file startup behavior, analysis branch throttling, initial frame publication policy

### Outcome B: worktree matches analysis ingress but is slower before first classification

Interpretation:

- regression is likely on the `health-falld` side
- examples: first-result policy, sequence warm-up requirement, startup buffering, model invocation strategy

### Outcome C: both stages are slower

Interpretation:

- regression is cross-service
- optimization must be split across producer and consumer boundaries

### Outcome D: worktree is not worse than `main`

Interpretation:

- the subjective lag report may come from a separate perception issue
- examples: UI/result alignment, overlay update cadence, result persistence policy
- in that case the next debugging step should move to presentation synchronization rather than startup classification latency

## 9. Out of Scope

The following are explicitly out of scope for this design document:

- immediately redesigning the action/fall model
- changing sequence length or training strategy before the baseline gap is measured
- redesigning the UI preview path
- switching the UI to shared memory or DMABUF
- claiming a hard `<= 0.3s` target before the `main` baseline is known

Those may become later phases, but they are not part of this baseline-establishment step.

## 10. Planned Next Step

After this spec is approved, the next step is to create an implementation plan that:

- adds a repeatable latency measurement path for `main` and the worktree
- runs both builds on the RK3588 board with `~/Videos/video.mp4`
- records the first-classification startup latency and supporting breakdown metrics
- uses the resulting A/B data to define the minimal code change required to make the worktree not worse than `main`
